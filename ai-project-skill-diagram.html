产出：project-spec.md
01
需求澄清背景 · 目标 · 约束 · 边界
02

04

⬇ project-spec.md 广播给所有角色

A

B

🛡️
QA（质量门禁 Agent）
Phase 3 · 强制验收 · 按协议执行 · 不自己发明验收方法
Heartbeat 自动触发
✅ 验收通过条件（PM 阶段定义，不可修改）
  • 前端：div depth = 0（Python 自查脚本）
  • 前端：浏览器截图内容正确（不用 headless）
  • 后端：接口返回 row_count > 0
  • 内容：KM 文档 > 100 字符
  • Cron：openclaw cron list 确认注册
  • git：push 成功，commit 符合预期
↩ 打回 / 升级规则
  • 验收不通过 → 打回对应 Worker 重做
  • 同一问题 retry ≥ 2 → 标记 failed
  • failed → 错题本记录 + 通知用户
  • 打回的是 Worker，不是用户
  • 错误复发 ≥ 3 次 → 自动升 P0 规则
通过 → TL 汇总 → 通知用户
打回 → Worker 重做（最多 2 次 retry）
超限 → failed + 错题本 + 用户介入
⬇ QA 全部通过
🚀
交付（TL 汇总 → 用户）
Phase 4 · 成本报告 + 关闭项目
用户收到最终交付物
📦 代码 / 页面已部署
📄 文档 / Wiki 已创建
⏱️ Cron 已注册且运行正常
📊 成本报告（token 消耗）
📚 错题本更新（防止复发）
🔒 铁律（从 quietview 项目踩坑中提炼）
1
执行前必须备份，备份确认存在才能动手
2
所有执行角色必须读 project-spec.md 里的架构约束
3
验收协议在 PM 阶段定好，QA 只能用协议里的方法
4
质量门禁是强制的，不是可选的
5
主 Agent 只做调度，不做执行，保持响应速度
6
Worker 完成 = 提交 review_needed，不是告诉用户完成了
7
截图验收必须用真实浏览器（不用 headless Chromium）
8
错误复发 ≥ 3 次自动升 P0，写入错题本防止下次重犯
