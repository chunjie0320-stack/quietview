# 搜索demo算法逻辑整理

> Demo 文件：`persona_demo_v5.html`（画像驱动的个性化搜索 Demo）

---

## 一、这个 Demo 解决什么问题？

**搜同一个词，为什么不同的人应该看到不同的结果？**

传统搜索的逻辑是：给所有人返回相同的排序。但现实中，不同用户的诉求差异巨大——

- 健身人群搜「沙拉」，最想看高蛋白低卡的
- 学生党搜「沙拉」，最想看价格便宜的
- 商务人士搜「沙拉」，最在乎品质口碑

**这个 Demo 验证了一件事**：把用户画像引入搜索排序，让搜索结果真正因人而异，而不只是全局热度榜。

## 二、Demo 具备哪些能力？

### 能力 1：个性化排序 —— 同词不同序

**是什么**：相同的搜索关键词，不同用户画像，返回不同的排序结果。

**解决什么问题**：传统搜索对所有人展示同一套结果，忽视了用户个体差异。本能力让排序"懂用户"，价格敏感的用户优先看到便宜的，健康偏好的用户优先看到健康分高的。

**效果示例**（搜索「沙拉」）：

| 商品 | 小美（健康优先）| 阿强（价格优先）|
|------|------|------|
| 牛油果鸡胸肉沙拉（¥48，健康分极高）| 🥇 第1 | 🥉 第3 |
| 蔬菜水果沙拉（¥32，价格适中）| 🥈 第2 | 🥇 第1 |
| 帝王蟹沙拉（¥88，品质顶尖）| 🥉 第3 | 🥈 第2 |

---

### 能力 2：首页个性化推荐 —— 纯画像驱动，无需关键词

**是什么**：首页「为你推荐」模块，不依赖任何关键词输入，直接根据用户画像从全量商品中打分，取 Top 8 展示。

**解决什么问题**：用户进入首页时还没有明确意图，传统方案只能展示通用热销榜。本能力让首页推荐"认识用户"，健康用户的首页与价格敏感用户的首页完全不同。

**使用场景**：用户打开 App 首页，无需任何操作，就能看到符合自己口味和偏好的商品推荐。

---

### 能力 3：实时行为捕捉 —— 点击偏好即时反馈

**是什么**：用户每次点击商品，系统实时统计行为，即时推断用户当前偏好倾向。

**解决什么问题**：用户的兴趣是动态变化的，静态画像无法反映当下诉求。本能力让系统感知"此刻"的用户偏好，点了几次低价商品就标注"价格敏感"，点了某类品类就积累品类偏好信号。

**使用场景**：用户在会话中连续点击了3次健康轻食，系统立即识别偏好并体现在后续推荐结果中。

> 注：当前 Demo 为内存态，刷新后重置。生产环境需对接行为日志系统实现持久化。

---

### 能力 4：接入真实 Persona 标签 —— 画像标签可视化

**是什么**：切换不同 Demo 用户时，系统调用美团 Persona 平台真实接口，拉取该用户的画像标签，并展示在「Persona 画像标签」面板中。

**解决什么问题**：让合作方直观看到"我们的用户画像长什么样"，不是抽象的 0/1 权重，而是真实可读的标签描述（如"健康偏好用户"、"价格敏感人群"等）。

**使用场景**：向合作方演示时，切换不同用户身份，现场拉取 Persona 真实标签，展示画像数据的可用性和覆盖度。

> 注：需登录 persona.sankuai.com 后才能获取真实数据。

---

### 能力 5：跨用户对比 —— 直观呈现个性化效果差异

**是什么**：搜索后自动生成「跨用户对比」面板，将同一关键词在4个不同画像用户下的 Top3 结果并列展示。

**解决什么问题**：个性化效果"说起来好，看不出来"。跨用户对比面板让差异一目了然，同一个搜索词，四个人的结果排列完全不同，一屏直观呈现画像驱动的价值。

**使用场景**：向合作方或业务团队演示个性化价值时，直接用对比面板说话，无需解释算法，结果差异一眼可见。

## 四、技术实现（附录）

> 以下为算法技术细节，供开发/算法同学参考，不影响理解 Demo 能力。

### 4.1 整体架构

```
用户画像（Persona标签 + 偏好权重）
        ↓
  关键词召回（多字段匹配）
        ↓
  个性化排序（scoreProduct 打分）
        ↓
  结果展示 + 跨用户对比
```

### 4.2 用户画像模型字段

| 字段 | 说明 | 取值范围 |
|------|------|---------|
| `priceWeight` | 价格敏感度（越高越倾向低价）| 0.0 ~ 1.0 |
| `healthWeight` | 健康偏好（越高越倾向低卡/高蛋白）| 0.0 ~ 1.0 |
| `qualityWeight` | 品质偏好（越高越倾向高评分）| 0.0 ~ 1.0 |
| `categoryBonus` | 对特定品类的额外加成（来自历史行为偏好）| 0.0 ~ 0.3 |
| `labelQueries` | Persona 真实标签查询词（用于从画像接口拉取标签）| 字符串数组 |

### 4.3 召回逻辑

对全量商品做多字段模糊匹配，命中任一条件即纳入候选集：

```javascript
const matched = PRODUCTS.filter(p =>
  p.name.includes(kw)          // 商品名包含关键词
  || p.category.includes(kw)   // 品类匹配
  || p.searchKeys.some(k => k.includes(kw) || kw.includes(k))  // 搜索词双向包含
  || p.tags.some(t => t.includes(kw))  // 标签匹配
);
```

- 双向包含匹配，兼顾短词和长词
- 多字段并联，召回率优先
- 当前为内存全量扫描，适合 Demo 场景（生产建议切换为倒排索引/ES）

### 4.4 个性化排序打分公式

```javascript
function scoreProduct(product, user) {
  let s = product.baseScore;

  // 健康加分：用户健康偏好 × 商品健康分 × 系数0.3
  s += product.healthScore * user.healthWeight * 0.3;

  // 价格惩罚：价格档位越高，价格敏感用户扣分越多
  s -= (product.priceLevel - 1) * user.priceWeight * 0.25;

  // 品质加分：用户品质偏好 × 商品品质分 × 系数0.2
  s += product.qualityScore * user.qualityWeight * 0.2;

  // 品类加成：来自用户历史行为的品类偏好
  s += (user.categoryBonus[product.category] || 0);

  // 随机扰动：防止排序完全固定，模拟真实噪声
  s += (Math.random() - 0.5) * 0.06;

  return Math.max(0, Math.min(1, s));  // 截断到 [0, 1]
}
```

**各因子权重对比：**

| 因子 | 最大贡献 | 说明 |
|------|---------|------|
| baseScore | ~0.82 | 商品基础热度，占主导 |
| 健康加分 | +0.285 | 健康重度用户对高健康分商品提升明显 |
| 价格惩罚 | -0.50 | 极度价格敏感用户对高端商品大幅降权 |
| 品质加分 | +0.19 | 相对较小，作为辅助因子 |
| 品类加成 | +0~0.30 | 强个性化信号，上限最高 |
| 随机扰动 | ±0.03 | 引入轻微随机性 |

### 4.5 Persona 标签接口

```
GET https://persona.sankuai.com/api/v2/ai/rag/dutvs/assets/searchAll
  ?query=<labelQuery>
  &assetTypes=label
  &idType=1
```

每个用户有 2~3 个 `labelQueries`，并行拉取，结果展示在「Persona 画像标签」面板。

### 4.6 扩展方向（Demo → 生产）

| 模块 | 当前实现 | 生产建议 |
|------|---------|---------|
| 召回 | 内存全量遍历 | 倒排索引 / ES |
| 排序 | 线性加权公式 | LTR（Learning to Rank）模型 |
| 用户画像 | 静态预设权重 | 实时行为流计算 |
| 品类加成 | 硬编码 | 从点击/购买日志实时计算 |
| 随机扰动 | 固定幅度 ±0.03 | ε-greedy 探索策略 |
| A/B测试 | 无 | 按用户分桶实验不同权重 |
